# Homiquity Product Spine

**Mission:** Clarity for every stage of homeownership.
**Core promise:** Organized, lender-ready mortgage packages — fast, honest, compliant.

---

## Data Flow (The Critical Path)

```
Borrower Input --> Coach Intake --> Borrower Graph --> Borrower Package --> Underwriting Review
     ^                ^              ^                  ^
  Application    AI Conversation   Documents       Deterministic
    Form         (structured       (OCR +           Rules Engine
                  extraction)      classification)
```

**Borrower Graph** is the unified read layer — it aggregates data from applications, documents, coach conversations, goals, and activity into a single queryable profile per borrower. Modules write to their own data stores; the graph computes across all of them.

## Borrower Package (11 Sections)

The lender-facing output. Generated by Coach, displayed in `BorrowerPackageView`.

1. **Borrower Overview** — Names, household composition, residence state, income profile type
2. **Household Details** — First-time buyer status, veteran status
3. **Transaction Intent** — Transaction type, property intent, target timeframe
4. **Declared Income Summary** — Source, type, frequency, documentation status (no amounts)
5. **Asset Summary** — Asset type, account category, ownership, documentation, last statement date, validation notes, access link (no balances)
6. **Credit and Debt Signals** — Score, monthly debts, DTI ratio with verification tiers
7. **Property Context** — Address, estimated value/price, occupancy intent
8. **Document Inventory** — Document type, label, status, and validation flags
9. **Readiness Status** — Intake status, document status, package status, pending items (procedural only)
10. **Audit Trail** — Intake start date, last update, timestamped activity log
11. **Validation Notes** — Recency checks, completeness checks, consistency observations (neutral language only)

Compliance footer on every package. No approval language. No eligibility assessment.

## AI Boundaries (Non-Negotiable)

| AI **does** | AI **never** does |
|---|---|
| Extract data from documents | Make lending decisions |
| Structure intake conversations | Imply approval or eligibility |
| Build borrower packages | Assess risk or creditworthiness |
| Flag procedural gaps | Recommend specific loan products as decisions |
| Surface eligible loan types from borrower data | Guarantee eligibility or approval for any program |
| Record factual observations | Use qualitative language about borrower strength |

All underwriting decisions are deterministic, rules-based, and auditable.

## Technology Stack

- **Frontend:** React 18, TypeScript, Vite, Wouter, TanStack Query, Shadcn/ui, Tailwind
- **Backend:** Node.js, Express, TypeScript, Drizzle ORM
- **Database:** PostgreSQL (Neon serverless)
- **AI:** OpenAI (Coach conversations), Google Gemini (document extraction)
- **Auth:** Replit OIDC + Passport.js, role-based access
- **Storage:** Replit Object Storage (documents)
- **Verification:** Plaid (employment, income, assets, identity)

## What "Done" Looks Like

A borrower can: talk to Coach, upload documents, get a structured package, receive a pre-approval letter — all in one session.

A lender can: open the package, see every document organized, click to view each one, know exactly what's missing, make a decision fast.
