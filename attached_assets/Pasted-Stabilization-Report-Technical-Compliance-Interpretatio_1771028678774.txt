Stabilization Report — Technical & Compliance Interpretation
Below is a clean, authoritative interpretation of each issue you surfaced — what it means, why it matters, and the underlying principle it violates.

1. CoachingProfile contains prohibited fields
Issue:  
recommendedLoanTypes, strengths, summary invite qualitative or eligibility‑adjacent language.

Why it matters:  
These fields force the AI into prohibited territory. Even if the system prompt says “never recommend products,” the schema itself contradicts that rule. Schema always wins.

Principle violated:  
AI must not recommend loan products, assess borrower quality, or produce qualitative judgments.

Required fix:  
Remove these fields entirely or rename them to procedural equivalents (e.g., “completedInputs,” “missingInputs,” “profileNotes”).

2. Borrower Package Builder prompt is stale vs. schema
Issue:  
The prose description (Section 6) and the JSON schema template describe different structures.

Why it matters:  
Conflicting instructions produce nondeterministic output. The AI will oscillate between the two formats depending on token weighting.

Principle violated:  
A single source of truth for output structure.

Required fix:  
Delete the prose description. Keep only the JSON schema. The AI should never be given two competing definitions of the same artifact.

3. “ready_now” tier uses underwriting thresholds
Issue:  
DTI < 43%, credit 680+, stable employment, etc.

Why it matters:  
These are eligibility criteria, not procedural readiness criteria. This crosses the compliance line into underwriting inference.

Principle violated:  
Readiness = completeness of inputs, not creditworthiness.

Required fix:  
Redefine readiness tiers strictly around:

profile completeness

document completeness

validation completeness

package assembly completeness

No ratios, no scores, no thresholds.

4. AI cannot generate valid document URLs
Issue:  
accessLink on assetSummary expects real URLs.

Why it matters:  
The AI cannot know your document IDs or storage paths. It will hallucinate URLs or leave them blank.

Principle violated:  
AI must not fabricate system‑internal identifiers.

Required fix:  
Remove accessLink from AI‑generated fields. Populate it server‑side after document ingestion.

5. CoachingProfile.summary has no compliance guardrail
Issue:  
Free‑text summary invites qualitative borrower assessment.

Why it matters:  
Even mild language (“strong borrower,” “solid income”) is prohibited.

Principle violated:  
AI must not characterize borrower quality.

Required fix:  
Rename or constrain the field to something procedural, e.g., “profileStatusNotes.”

6. readinessScore (0–100) looks like approval probability
Issue:  
Numeric score invites misinterpretation.

Why it matters:  
Borrowers will assume it reflects approval likelihood, even if the prompt says otherwise.

Principle violated:  
No numeric scoring that resembles risk, eligibility, or probability.

Required fix:  
Replace with categorical procedural states (e.g., “profile_complete,” “documents_pending,” “package_ready”).

7. Veteran status annotated as “VA loan eligible”
Issue:  
The code explicitly states an eligibility conclusion.

Why it matters:  
This is a direct underwriting determination.

Principle violated:  
AI and system must not infer product eligibility.

Required fix:  
Output only: “Veteran Status: Yes/No.”

8. Property context prompt asks AI to assess financial suitability
Issue:  
“Assess whether the user’s financial profile supports this purchase price.”

Why it matters:  
This is underwriting analysis.

Principle violated:  
AI must not evaluate affordability or suitability.

Required fix:  
Remove the instruction entirely. Property context should be factual only.

9. No schema validation on AI JSON output
Issue:  
JSON.parse + key checks = silent corruption.

Why it matters:  
Malformed AI output will partially render, break UI, or create inconsistent state.

Principle violated:  
AI output must be validated before use.

Required fix:  
Add Zod schemas for:

CoachingProfile

BorrowerPackage

ReadinessState

DocumentInventory
Reject or request regeneration on mismatch.

10. formatCurrency silently corrupts values
Issue:  
Number("") becomes 0, not NaN.

Why it matters:  
“Not Provided” → $0 is a catastrophic misrepresentation.

Principle violated:  
Never coerce non‑numeric text into numeric values.

Required fix:  
Explicitly check for numeric strings before formatting.

11. No error boundary on BorrowerPackageView
Issue:  
Unexpected shapes crash the component.

Why it matters:  
AI output is inherently probabilistic; UI must be defensive.

Principle violated:  
Never trust AI output without guardrails.

Required fix:  
Wrap the entire package view in an error boundary with fallback UI.

12. “strengths” and “gaps” terminology
Issue:  
These are qualitative assessments.

Why it matters:  
They imply borrower quality and improvement areas.

Principle violated:  
AI must not coach toward eligibility.

Required fix:  
Rename to:

“completedInputs”

“missingInputs”

13. Stall response uses “strong foundation”
Issue:  
Qualitative borrower assessment.

Why it matters:  
Even soft praise is prohibited.

14. Stall response uses “solid progress”
