1. High‑level workflow
Stage 1 — Lead / Education / Soft Prequal  
Soft pull (or soft score) during initial intake and education. No hard credit impact. Capture consent for communications and a separate, explicit consent checkbox for any future hard pull.

Stage 2 — Intent and Commitment  
When borrower identifies a property and submits an offer (or signs a purchase agreement), mark the file as “Committed.” At this point require a signed application or a small refundable deposit (optional) to demonstrate intent.

Stage 3 — Tri‑merge Hard Pull  
After commitment and signed consent for a hard pull, execute tri‑merge. Generate AUS/underwriting package and proceed to lender submission. Log adverse‑action triggers and generate notices if needed.

2. Consent, disclosures, and audit requirements
Two consent events must be captured and stored immutably:

Soft‑pull consent (marketing/prequal; non‑FCRA hard pull).

Hard‑pull consent (explicit FCRA language authorizing a credit report for underwriting).

Store: name, email, timestamp, IP, user agent, exact consent text, and a hash of the consent record.

Audit trail: append‑only event store for consent_given, soft_pull, commitment_received, hard_pull_requested, hard_pull_completed, adverse_action_sent. Exportable CSV/PDF for exams.

3. Replit implementation checklist (endpoints and events)
Frontend actions

POST /api/lead → create lead; trigger viewed_education and soft_pull_opt_in.

POST /api/soft-pull → run soft pull via vendor; store soft result; event soft_pull_completed.

POST /api/commit → upload purchase agreement or mark offer submitted; store commitment artifact; event commitment_received.

POST /api/hard-pull-consent → capture signed FCRA consent; event consent_hard_pull.

POST /api/hard-pull → server‑side tri‑merge call; event hard_pull_completed.

Backend responsibilities

Validate consent before any hard pull.

Use idempotency keys for credit API calls.

Store raw bureau response IDs and parsed fields separately.

Generate and store adverse action notices automatically when bureau response triggers denial or adverse decision.

Events to log

start_prequal, complete_prequal, download_checklist, reserve_seat, commitment_received, hard_pull_requested, hard_pull_success, hard_pull_failure, adverse_action_generated.

4. Cost control and monetization options
Soft pulls for all leads to reduce early costs.

Tri‑merge only after commitment.

Fee models (choose one or combine):

No fee, staged approach: absorb soft pull cost; tri‑merge billed to broker and recouped via lender compensation.

Application/credit fee: small, disclosed fee to cover tri‑merge; refundable or credited at close.

Refundable deposit: refundable at close; forfeited if borrower withdraws without valid reason.

Operational rule: never run tri‑merge for low‑intent leads; require signed application or purchase agreement.

5. Adverse action automation and compliance
Adverse action engine maps bureau reason codes to model adverse notices.

Trigger points: denial by AUS, borrower withdraws after tri‑merge, or credit score drop causing denial.

Delivery: generate PDF notice, email and store copy in audit log, record delivery timestamp and method.

Retention: keep adverse action records and all credit pull logs for at least 3 years.

6. KPIs and monitoring
Prequal → Commitment rate (target > 20% initially)

Commitment → Tri‑merge rate (should approach 90% if gating is strict)

Tri‑merge cost per funded loan (track to ensure profitability)

Stranded pull rate (tri‑merge pulls that do not convert to application/close) — aim to minimize

Time from commitment to hard pull (target < 48 hours)

Adverse action frequency and reasons (use to improve credit‑readiness content)
